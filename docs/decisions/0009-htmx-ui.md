# ADR-009: HTMX-based UI in v1 rather than a React SPA

**Status:** Accepted
**Date:** 2026-05-16
**Source:** Implementation choice

## Context

The design doc calls for a Web UI as a Phase 1 deliverable: project list,
run detail, leaderboards. The implementation plan also notes "no dedicated
frontend headcount in Y1" — UI work rotates onto a backend engineer for a
sprint.

A React/Vite SPA needs a build pipeline, a state-management story, an API
client, and ongoing dependency upkeep. For a six-page admin UI consumed by
internal engineers, that's a lot of moving parts to justify.

## Decision

- **Server-rendered HTML with HTMX for interactivity.** Jinja2 templates in
  `melp/services/ui/templates/`; one CSS file at `static/melp.css`; HTMX from
  a CDN script tag.
- **The UI service reads directly from Postgres** via the same SQLAlchemy
  models as the API services. No separate API client, no JSON parsing,
  no state synchronisation.
- **Pages in v1:** home (projects), project (runs list), run (status,
  progress, per-slice results with CIs).
- **Migration path to React/Vite is open.** The HTML is generated by a
  separate FastAPI service (`melp.services.ui.app`); swapping it for an SPA
  later doesn't touch any other service.

## Consequences

- **One backend engineer can ship and maintain this** in a few days. No
  frontend toolchain to learn.
- **The UI's contract is the database schema.** That's a benefit (no schema
  drift between server and client) and a constraint (no offline state, no
  optimistic updates).
- **Limits on richness.** Real-time updates use HTMX polling, not WebSockets.
  Complex visualisations (interactive plots) would force a different choice.
  Acceptable for v1 — the dashboards in §13.4 are tables and time-series, not
  interactive analysis tools.
- **Some users will want React anyway.** When a real frontend team picks this
  up, they replace the UI service wholesale. The decision is reversible.

## Alternatives considered

1. **React + Vite SPA.** Right answer eventually, wrong answer now. The
   total cost of ownership through Phase 2 is higher than HTMX, and the
   features it unlocks (rich interactivity, custom viz) aren't ones we need
   in v1.
2. **Streamlit / Gradio.** Faster to prototype, harder to make look
   professional or theme consistently. Not built for permanent admin UIs.
3. **Skip the UI entirely; CLI-only.** Plan's cut list includes this as a
   tier-1 cut if we slip. We didn't slip; HTMX is cheaper than no UI at all
   because pilot teams meaningfully prefer a browser to a terminal for
   reviewing results.
