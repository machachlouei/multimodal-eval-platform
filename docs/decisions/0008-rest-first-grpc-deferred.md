# ADR-008: REST/FastAPI first; gRPC deferred

**Status:** Accepted
**Date:** 2026-05-16
**Source:** Implementation choice; divergence noted from Design-Doc.md §8.1

## Context

Design-Doc.md §8.1 lists three protocols on the public surface: REST, gRPC,
and internal gRPC. With one lead + four engineers + one DS in Year 1, "ship
all three at launch" is unrealistic. We have to pick one to lead with.

## Decision

- **REST/JSON over HTTPS is the only public protocol in v1.** Documented via
  OpenAPI auto-generated by FastAPI; surfaced at `/docs` for every service.
- **Python SDK wraps REST.** No protobuf code-gen needed for the canonical
  client.
- **CLI wraps the SDK.** Single source of truth for client behavior.
- **Internal service-to-service is plain HTTP over the cluster mesh** (mTLS
  provided by the mesh, not the application). Switching to gRPC later is a
  per-service migration, not a platform-wide cutover.
- **gRPC re-evaluated at the end of Phase 2** if a real user team is paying
  for the latency difference. Likely candidates: the judge orchestrator
  (per-example fan-out at scale), the metric pool (high-rate result writes).

## Consequences

- **Faster time-to-pilot.** No protobuf build pipeline, no dual-language
  client maintenance.
- **OpenAPI is the contract.** Generated, versioned, and used by SDK
  consumers and CI integrations directly.
- **Latency-sensitive paths pay a JSON-parsing cost.** Acceptable at Year-1
  scale (2,000 runs/day per §11.1). Will likely matter at Year-3 scale for
  the judge orchestrator's per-example fan-out — that's the trigger to
  revisit.
- **Spec divergence is documented.** §8.1 calls out gRPC; we explicitly
  deferred it. The doc and the code agree once this ADR is in the index.

## Alternatives considered

1. **gRPC + REST gateway from day one.** Faster for hot paths, slower for
   everything else (engineer ramp-up, debugging, SDK ergonomics). Not worth
   it at pilot scale.
2. **REST only, ever.** Likely fine forever for the control plane; not fine
   forever for the judge orchestrator at scale. Better to leave the door
   open.
3. **GraphQL.** Wrong shape — the API is verb-heavy (submit run, cancel run,
   publish dataset) rather than query-heavy. GraphQL's strengths don't apply.
